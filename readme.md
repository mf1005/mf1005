## 🧩 エンジニアとしての原点：10KBの極限下で磨いた「最適化のDNA」
**「仕様がないなら解析し、繋がっていないなら技術で繋ぐ」**

JOYSOUND、DAM、UGAなど、メーカーごとに非公開だったカラオケ機器の赤外線通信プロトコルを独学で解析。
カラオケボックスにPCと赤外線解析機を自ら持ち込み、**「歌わずに信号をノートに記録し続ける」という地道な現場解析**を何度も繰り返しました。メーカーごとに異なるバイナリ信号のパターンを読み解き、通信フォーマットを特定する。この **「仕様がないなら、実機から聞き出せばいい」** という姿勢が私の原点です。

当時のiアプリの容量制限はわずか**10KB**であり、その中にマルチメーカー対応のカラオケリモコン機能を凝縮させました。
インターフェースによる抽象化すら容量を圧迫する極限状態の中、綺麗な設計という贅沢を捨て、ベタ書きのロジックのみで複数メーカーの通信プロトコルを完遂。

単なるアプリに留まらず、自作の「楽曲検索サイト」から直接iアプリを起動し、そのまま赤外線で選曲したり、「お気に入りソング」をDB保存してどこでも呼び出せる仕組みを構築。今では当たり前の **「Webとデバイスをシームレスに繋ぐ体験」** を、20年前に独力で形にしていました。

※その後、スマホへの移行とデバイスからの赤外線ポート消失に伴い、本プロジェクトは技術的使命を終えましたが、ここで得た知見は今も私の血肉となっています。

#### **【深掘りエピソード】10KBの壁を無効化した「インストールさえ通れば勝ち」のハック**

  現在で言うところの「アセットバンドル」や「オンデマンド・リソース」の概念となります。物理的に不可能と思われた「10KBでのマルチメーカー対応」を実現したこの経験が、私のハッカー精神の原点です。

改善試行: ページネーションを復活させるため、ユーザーアクセスをトリガーにDB結果をキャッシュ化することで、以前と同じUIを維持に成功。  
新たな問題: キャッシュ切れの瞬間に大量のリクエストがDBへ殺到。キャッシュのためのSQLのスロークエリが多発し、二度目の停止警告を受ける。(その際の受信メールを後述しているので参照）

解決策: ユーザーのリクエスト時にDBで集計を行うのをやめ、集計済みデータオブジェクトのキャッシュから取得するようデータソースの完全切り替えを実施。

従来の「ユーザーアクセスをトリガーとするキャッシュ生成」を廃止し、<strong>「バッチによる完全非同期・先行生成」</strong>と<strong>「イベント駆動の差分更新」</strong>へ刷新することで、参照系のDB負荷を極限まで減少。

**■ 整合性担保のロジック**

フラグ管理: バッチ開始時にフラグを立て、**現在、キャッシュ生成中である**ことをシステム全体に通知。  
楽観的更新: フラグが立っている間のユーザー投稿は、DBへの保存とは別に、古いキャッシュファイルに対しても暫定的な追記（Diff Merge）を行い、表示上の即時反映を維持。  
事後マージ: Cronは「ベースとなるキャッシュ」を作り終えた後、「バッチ開始時刻 〜 現在」の間に更新されたレコードを再度DBから取得。最後にこれをベースキャッシュにマージすることで、データ欠損を防止。

### 🛠 技術的深掘り：整合性を死守するキャッシュ戦略

月間120万PVを超える高負荷環境において、レンタルサーバーの制約を打破するために構築した、独自のキャッシュ整合性アルゴリズムです。

#### 1. 課題：バッチ実行中の「データ先祖返り」リスク
大規模な事前キャッシュ生成バッチ（約10〜20分実行）を行う際、以下の不整合が課題となりました。
- **バッチ開始時**のDB状態を元にキャッシュを作ると、**バッチ実行中**にユーザーが投稿した最新データが、バッチ終了時のキャッシュ上書きによって消えてしまうロストアップデート問題。

#### 2. 解決策：二段構えの「差分マージ」アーキテクチャ
レンタルサーバで稼働しているシステム、かつ、Redisといった高機能なメッセージキューを使わない制約下で、アプリケーションレイヤーでの整合性舗装を行いました。

- **Phase 1：実行フラグによる動的ルーティング**  
  バッチ開始時に `process_flag` をオンに設定。この間、ユーザーの更新処理（`ReviewService`）は、通常のキャッシュ更新に加え、 **「差分管理テーブル（review_diff）」** にも最新レコードを退避させます。
- **Phase 2：ラスト・マージによる最終整合**  
  ベース構築（Step A）が完了した直後に、バッチ実行中に溜まった `review_diff` のみを再スキャン（Step B）。ベース構築済みのキャッシュに対して最新状態を「上書きマージ」することで、データの整合性を担保しました。

#### 3. 🛡 競合状態と整合性への設計思想（戦略的トレードオフ）

本プロジェクトでは、厳密な排他制御を導入せず、**「ビジネス上の許容範囲」と「エンジニアリングのROI」を天秤にかけた戦略的トレードオフ**を選択しています。

- **「直し方は把握しているが、あえて直さない」という意思決定**  
  現在のアーキテクチャでは、夜間バッチ実行中のデータ削除の反映漏れや、ミリ秒単位の同時更新における情報の欠落（レースコンディション）の可能性があることを把握しています。
  これらは削除情報の差分管理や排他制御の導入で技術的に解決可能ですが、以下の理由からあえて「対応しない」という戦略的判断を下しています。

  - **リスクベースのROI評価：** 発生確率が極めて低く、実害が軽微かつ翌日の夜間バッチで自動修復されるケースに対し、**修正コードの投入によって予期せぬ不具合（デグレ）やサイト停止を招くリスク**を天秤にかけました。結果として、現在の「シンプルかつ安定した構造」を維持することが、事業継続において最も価値が高いと判断しています。
  - **ランタイムの複雑性回避：** 共有ホスティング環境の制約下において、複雑な排他制御を導入することはデッドロックやパフォーマンス低下の温床となる可能性を否定できません。ステートレスに近い挙動を維持することで、可用性という最大のユーザー体験を死守しました。

- **「24時間以内の自己修復（Self-healing）」プロトコル**  
  日中の微細な不整合を許容する大前提として、**デイリー夜間バッチによる全件キャッシュ再生成**を組み込んでいます。
  - 万が一不整合が生じても、最大24時間以内に必ずクリーンな状態へ同期される「自浄作用」をシステムに持たせることで、運用コストを最小化。
  - 100%の整合性を維持するためにシステムを複雑化させるのではなく、「翌日には必ず直っている道（舗装）」を敷くことで、低コスト・高可用性・保守性の三権分立を成立させました。

> **【舗装屋の視点】** 
> 完璧なシステムを追求して事業を鈍化させるのではなく、「どこまで手を抜いても死なないか」を正確に見極め、限られたリソースを事業成長の急所に投下する。これも、私が20年のキャリアで培ったエンジニアリングの誠実さの一環です。


#### 4. 実装のこだわり
- **メモリ管理の徹底**: `DB::disableQueryLog()` の実行や `unset()` によるメモリ解放を駆使。共有サーバーの厳しい制限下でも、メモリリークを抑えバッチを完遂させる実戦的なコードを徹底しました。
- **保守性の高い舗装**: 複雑な排他ロジックを排除し、シンプルで予測可能な挙動にすることで、Laravelを知っていればメンテナンスできる舗装されたコードを維持しています。

#### **【闘争のプロセス：事実に基づく科学的な切り分けと決断】**

社内インフラ部隊が「インフラ側に異常なし」とし、アプリ側のロジックを疑う声が強まる中、私は「事実（ファクト）のみを信じる」スタンスを貫き、以下のロジックで原因を特定しました。

1. **環境比較による原因の切り分け（変数の特定）**:
    * **ローカル環境（正常）**：アプリの認証ロジック、コードそのものに不備はないことを確認。
    * **GCE：常駐インスタンス（正常）**：GCP上のDB、ネットワークに問題がないことを確認。
    * **Cloud Run：サーバーレス（異常発生）**：異常が起きるのはCloud Run環境のみ。つまり、サーバーレス特有の **「インスタンスの揮発性（消滅と生成）」および「オートスケール」** に原因があると仮説を立てる。

2. **執念の「F5連打」による仮説検証**:
    この仮説をインフラ部隊に説明したところ、「最小インスタンス数=1」の設定変更で解決すると断言された。しかし、本質的な解決になっているかを確認するため、自らブラウザで **F5連打による負荷再現** を実施。
    * **検証結果**：オートスケールにより2台目が立ち上がった瞬間にログアウトが発生する事実をメトリクスから突き止める。根本原因は「最小インスタンス数」の設定といった小手先の小細工では解決不能であることを物理的に証明。

3. **戦略的トレードオフによる技術選定（Redisの採用）**:
    解決にあたっては、セッション情報を外部化する「完全ステートレス設計」への刷新を即断。極限のタイムリミットの中で以下の比較検討を行いました。
    * **検討した選択肢（Firestore）**：公式ドキュメント等では中小規模向けのセッションストアとして推奨されるが、本番リリース3日前という状況では未知のリスクを排除すべきと判断。
    * **採用した選択肢（Cloud Memorystore for Redis）**：Spring Sessionとの親和性が極めて高く、最も確実に、かつ最短で実装・検証を完遂できる手段として採用を決定。
    * **最適解**：Spring Bootのエコシステムをフル活用することで、**わずか数行の実装で堅牢な基盤への刷新を実現**。自身の技術的エゴではなく、プロジェクトの完遂とリリース後の保守性を最優先しました。

**結果：**
致命的な不具合を解消し、一度の遅延もなくSaaSサービスをリリース。現在に至るまで、セッション起因のトラブルはゼロを継続しています。

**【技術的意思決定の背景】**
私は「環境の違いで挙動が変わるのは、アプリケーション側の不備である」と考えています。本件においても、ローカル環境・GCE・Cloud Runの3環境を比較検証し、サーバーレス特有の変数を特定するに至りました。インフラ側のスペック調整ではなく、Redisを用いたステートレスなアーキテクチャへの刷新を選択することで、環境に依存しない堅牢なシステムへと昇華させました。

#### **【決断：リリース3日前のアーキテクチャ刷新と本番稼働後の戦略的撤退】**

- **インフラの正攻法（舗装）：**

#### **【ビジネス貢献】**

## **🤝 エンジニアとしてのスタンス**

#### **ビジネスへの向き合い方：「議論より、動くモックアップを」**
私は、「最新技術を使うこと」や「綺麗なドキュメントを書くこと」それ自体を目的にしません。私の価値は、ビジネスのフェーズに合わせて最適な「解」を出すことです。

- **0→1（検証）フェーズ：**
    会議室で10時間議論する暇があれば、その時間でコードを書き、動くものを見せて合意を取ります。ここではドキュメントよりもスピードと「動く事実」を最優先します。
- **1→10（運用・拡大）フェーズ：**
    ビジネス価値が証明された後は、SIerで培った経験を活かし、保守性・可読性を考慮したドキュメント化と、標準規約に則ったリファクタリングを徹底します。「作って終わり」ではなく「長く稼働し続ける資産」に昇華させます。

#### **技術への向き合い方：「根治（最適化）」を優先するエンジニアリング**
クラウド全盛の現代において、リソース不足を「札束（スペックアップ）」で解決することは容易です。しかし、私は安易なスペックアップの前に「アプリケーションレベルで解決できることはないか？」を徹底的に問うスタイルを貫いています。

- **一時的な回避策としてのスペックアップ：** ビジネスの機会損失を防ぐためにまず「時間を買う」判断もありうると考えています。
- **根本解決としての最適化：** スペックアップと並行して、必ずボトルネックの正体（非効率なクエリ、設計ミス）を突き止め、改修し、一時的に上げたスペックを下げられないかを判断します。
この「対症療法」と「根治」を戦略的に使い分けることで、システムの信頼性とビジネスの利益率を両立させるのが私のエンジニアとしての誠実さです。

#### **技術への適応力：「言語」の壁を超えて問題を解決する**
炎上案件の支援（火消し）では、Java, C#, Nuxt.js など、プロジェクトごとに技術スタックが固定されており、未経験の言語環境に飛び込むことも日常飯事です。

しかし、私は言語の文法暗記に頼らず、PHPのチューニングで培った **「プログラミングの基礎原理（メモリ、通信、アルゴリズム）」** でコードを読み解くため、どのような言語でもロジックを把握し、改修を行うことができています。

- **既存案件（火消し）:** 未知の言語でもソースコードから仕様を逆算し、ボトルネックを特定・修正。
- **新規案件（特命）:** Python (Slack API) など、開発スピードとライブラリの充実度を考慮し、要件に最適な技術をゼロベースで選定。
「使える言語」に縛られず、「解決すべき課題」に合わせて即座に手札を増やすスタイルです。

#### **組織への向き合い方：「誠実な客観性」という武器**
私はプロジェクトに対して常に『誠実』でありたいと考えています。

支援として参画した際、現場のリーダーが疲弊し、危機感を感じつつも上層部へのエスカレーションができない状況を目の当たりにしました。私は **『支援者という客観的な立場』を戦略的に活用**し、現場の忖度や疲弊に飲み込まれることなく、あえて空気を読まない『幽霊担当者入りのWBS』を提示しました。
これは、単なる独断専行ではなく、疲弊した現場に代わって事実を突きつけることが、プロジェクトとメンバーを救う唯一の手段だと判断したからです。この『越境する力』こそが、私の考えるテックリードの責務です。

しかしながら、私の突破力は、周囲の協力があって初めて成立するものです。だからこそ、周囲への適切な相談と、透明性の高いエスカレーションを、技術スキルと同じくらい大切にしています。
最終的な意思決定に従う柔軟性と、目的達成のために粘り強く対話を続ける姿勢。
この両輪で、不確実性の高いプロジェクトを確実に完遂させます。

## 🛣️ 私が「舗装屋」を名乗る理由

これまで20年間、仕様なき解析、極限の最適化、炎上案件の救済など、数多くの「泥臭い現場」を突破してきました。その時、常に感じていたのは、 **「今この場面で、私でない誰かが事前に道を整えてくれていたら、チームはどれほど速く前に進めただろうか」** という切実な思いでした。

本来、エンジニアリングは、もっと創造的で、ワクワクするものであるべきです。
私自身、10KBの死闘やキャッシュ・スタンピードとの戦いは、その瞬間は苦しくとも「どう制約を突破するか」を考えている時間は最高に楽しく、思い通りに動いた時の達成感は格別でした。

一方で、現実のプロジェクトには、誰もやりたがらない、しかし誰かがやらなければ進めない「泥」が至る所に転がっています。

私は、自分が培ってきた技術力と執念を、自分一人のためではなく **「次に続くエンジニアが、迷わず、全速力で駆け抜けるための道を作る」** ために使いたいと考えています。

## ⚙️「使われにいく」テックリードとして

私が目指すのは、高みから指示を出すリーダーではありません。現場でバリバリと新機能を実装するメンバーに寄り添い、**「今、何に困っているか」「どこを改善すればもっとプロジェクトが加速するのか」を徹底的にヒアリングし、自ら泥の中に手を突っ込んで道を舗装する役目**です。
どんな些細なストレスでも、現場の「負」を吸い上げ、技術でそれを「無かったこと」にする。
**チームが最高速を出すための「最も使い勝手の良いパーツ」** として、全員が全力で走れる環境を整えることに注力したいと考えています。
