## 🧩 エンジニアとしての原点：10KBの極限下で磨いた「最適化のDNA」
**「仕様がないなら解析し、繋がっていないなら技術で繋ぐ」**

私のエンジニアリングの根底には、20年前の学生時代に培った「理屈を超えた解決への執念」があります。
当時取り組んでいたのは、**個人の携帯電話をカラオケの多機能リモコン（現在のデンモク）に変える**という挑戦でした。

#### **1. 仕様なき場所での現場解析：バイナリから仕様を読み解く**
JOYSOUND、DAM、UGAなど、メーカーごとに非公開だったカラオケ機器の赤外線通信プロトコルを独学で解析。
カラオケボックスにPCと赤外線解析機を自ら持ち込み、**「歌わずに信号をノートに記録し続ける」という地道な現場解析**を何度も繰り返しました。メーカーごとに異なるバイナリ信号のパターンを読み解き、通信フォーマットを特定する。この **「仕様がないなら、実機から聞き出せばいい」** という姿勢が私の原点です。

#### **2. 10KBの死闘：抽象化を捨て、実利を削り出す**
当時のiアプリの容量制限はわずか**10KB**であり、その中にマルチメーカー対応のカラオケリモコン機能を凝縮させました。
インターフェースによる抽象化すら容量を圧迫する極限状態の中、綺麗な設計という贅沢を捨て、ベタ書きのロジックのみで複数メーカーの通信プロトコルを完遂。
1バイトを削るためにデータ構造とアルゴリズムを徹底的に最適化し、機能を凝縮させました。

#### **3. Webとアプリを繋ぐUXの先取り：エコシステムの構築**
単なるアプリに留まらず、自作の「楽曲検索サイト」から直接iアプリを起動し、そのまま赤外線で選曲したり、「お気に入りソング」をDB保存してどこでも呼び出せる仕組みを構築。今では当たり前の **「Webとデバイスをシームレスに繋ぐ体験」** を、20年前に独力で形にしていました。

※その後、スマホへの移行とデバイスからの赤外線ポート消失に伴い、本プロジェクトは技術的使命を終えましたが、ここで得た知見は今も私の血肉となっています。

#### **【深掘りエピソード】10KBの壁を無効化した「インストールさえ通れば勝ち」のハック**

> **「10KBはパッケージの限界であり、システムの限界ではない」**

20年前のiアプリ開発において、最大の壁は「複数メーカー対応のための膨大な赤外線データ」でした。全メーカーのプロトコルを抱えると10KBに収まらず、インストールすらできませんでした。
この時、私が考えたのは **「機能を削る」** ではなく、 **「インストールさえ通してしまえば、あとは通信でなんとかなるはずだ」** という、極めてシンプルな突破口でした。

- **「ゲート」を潜り抜けるための逆転の発想：**
  調査を進め、「10KBの制限はjarパッケージのインストール時にのみ適用される」という仕様があることに気づきました。
  この気づきにより、赤外線送信の直前にデータをダウンロードしてきても機能は成立すると考えました（パケット代の問題はさておき）。
  「パッケージを小さく保ち、中身は後から流し込めばいい」という直感に基づき、実装を行いました。
- **解決策としての「動的ロード」：**
  初期パッケージには通信エンジンのみを実装。アプリ起動後に各メーカーの赤外線データをHTTP通信で差分ダウンロードし、ガラケー内部のストレージ領域（スクラッチパッド）へ動的に展開し、必要に応じて参照するアーキテクチャを構築しました。
- **成果：**
  現在で言うところの「アセットバンドル」や「オンデマンド・リソース」の概念となります。物理的に不可能と思われた「10KBでのマルチメーカー対応」を実現したこの経験が、私のハッカー精神の原点です。

#### **「どれだけ制約があっても、インストール（突破）さえできれば道は舗装できる」**
この時培った「仕組みの急所を突き、強引にでも完遂させる」思考は、現在の私の「不確実なプロジェクトを技術と行動でハックする」姿勢に直結しています。

### **② 【個人開発】高効率アーキテクチャによる事業利益率の最大化**
> - **月間120万PVを月額1,000円のインフラで維持し、営業利益率99%超を実現**

<details>

**役割:** 
企画、開発、運用（すべて1人）

**環境:** 
PHP (Laravel), MySQL (Shared Hosting), Linux, Cron

**実績:** 
月間120万PVを月額1,000円のインフラで維持。3年間メンテナンスフリーで稼働中。

**【課題：成長の裏で繰り返された「サーバー停止警告」】**
サービス拡大に伴い、共有ホスティング（Mixhost）の制約下で二度の深刻な負荷危機に直面しました。
この際、安易なクラウド移行（スペックアップ）で解決を急がず、「対症療法（金で時間を買う）」と「根治（コードの最適化）」を戦略的に使い分けることで、極限の利益率を実現しました。

**【闘争のプロセス：3段階のアーキテクチャ進化】**
単なる技術選定ではなく、ビジネス（UX/PV）とインフラ負荷の天秤をかけながら、以下のステップで最適解を削り出しました。

**Step 1：緊急回避（暫定的な全件表示への仕様変更）**
判断: Laravelの標準ページネーション機能が内部で自動発行する集計クエリ（COUNT(*)）が、50万件超のデータにおいては致命的なDB負荷（スロークエリ）の主因となっていたため、標準機能の利用を一時的に廃止。
対応: フレームワークのブラックボックスな挙動を避け、DB負荷の低いシンプルな「全件表示」に切り替え、サーバーの即時停止を回避。
結果と葛藤: 負荷は下がったが、データ過多でUXが崩壊し、ページ遷移がなくなることでPV（収益）も減少。エンジニアとして **「サービスを守るが、ビジネスを殺している」状況に強い危機感を抱く。**

**Step 2：第2の警告と「キャッシュ・スタンピード」の失敗**
改善試行: ページネーションを復活させるため、ユーザーアクセスをトリガーにDB結果をキャッシュ化。
新たな問題: キャッシュ切れの瞬間に大量のリクエストがDBへ殺到。スロークエリが多発し、二度目の停止警告を受ける。
教訓: 「ユーザーのリクエストを起点にDBを叩く」という Web開発の常識自体が、この規模と環境ではリスクであること。

**Step 3：最終解：集計済みデータオブジェクトによるリードモデルの構築**
解決策: ユーザーのリクエスト時にDBで集計を行うのをやめ、集計済みデータオブジェクトのキャッシュから取得するようデータソースの完全切り替えを実施。
技術的工夫: 夜間バッチ（Cron）でユーザーIDごとの口コミ件数や内容を事前に集計し、「口コミデータオブジェクト（JSON/配列等）」として永続キャッシュ化。
挙動: ユーザーアクセス時は、ユーザーIDをキーにキャッシュから直接オブジェクトを取得・展開するのみ。DBへの検索・集計クエリを完全に排除することで、DB負荷をゼロ化しつつ、物理的な限界に近いレスポンス速度を実現。
整合性の担保: 更新時は「イベント駆動による差分マージ」を行い、バッチ実行を待たずに最新の状態がキャッシュに反映される仕組みを構築。

### 🛠 技術的深掘り：整合性を死守するキャッシュ戦略

月間120万PVを超える高負荷環境において、レンタルサーバーの制約を打破するために構築した、独自のキャッシュ整合性アルゴリズムです。

#### 1. 課題：バッチ実行中の「データ先祖返り」リスク
大規模な事前キャッシュ生成バッチ（約10〜20分実行）を行う際、以下の不整合が課題となりました。
- **バッチ開始時**のDB状態を元にキャッシュを作ると、**バッチ実行中**にユーザーが投稿した最新データが、バッチ終了時のキャッシュ上書きによって消えてしまうロストアップデート問題。

#### 2. 解決策：二段構えの「差分マージ」アーキテクチャ
レンタルサーバで稼働しているシステム、かつ、Redisといった高機能なメッセージキューを使わない制約下で、アプリケーションレイヤーでの整合性舗装を行いました。

- **Phase 1：実行フラグによる動的ルーティング**
  バッチ開始時に `process_flag` をオンに設定。この間、ユーザーの更新処理（`ReviewService`）は、通常のキャッシュ更新に加え、 **「差分管理テーブル（review_diff）」** にも最新レコードを退避させます。
- **Phase 2：ラスト・マージによる最終整合**
  ベース構築（Step A）が完了した直後に、バッチ実行中に溜まった `review_diff` のみを再スキャン（Step B）。ベース構築済みのキャッシュに対して最新状態を「上書きマージ」することで、データの整合性を担保しました。

#### 3. 🛡 競合状態への設計思想
実務的な視点から、ユーザー更新とバッチ更新の競合については、厳密な排他制御（Distributed Lock等）を導入しない **「結果整合性」重視の設計** をあえて採用しています。

- **「数秒のズレ」を許容した全体最適の判断**
  
  本サービスは口コミ投稿型プラットフォームであり、金融システムのような厳密なトランザクション整合性よりも、 **「いつアクセスしても止まらない可用性」と「最速のレスポンス」** がユーザー体験に直結します。複雑なロック機構は、デッドロックのリスクやスループットの低下を招き、結果としてシステム全体の信頼性を損なう可能性があると判断しました。

- **構造的なリセット：デイリー夜間バッチによる担保**
  
  日中の「結果整合性」を許容する大前提として、**デイリー夜間バッチにて事前キャッシュを全件再生成する設計**としています。
  万が一、日中の差分マージにおいて微細な不整合が生じたとしても、最大24時間以内に必ず完全にクリーンな状態へ同期される**自浄作用**をシステムに持たせています。この多層的な設計により、ビジネス要件を満たしつつ、システム負荷を最小限に抑える全体最適を実現しました。

#### 4. 実装のこだわり
- **メモリ管理の徹底**: `DB::disableQueryLog()` の実行や `unset()` によるメモリ解放を駆使。共有サーバーの厳しい制限下でも、メモリリークを抑えバッチを完遂させる実戦的なコードを徹底しました。
- **保守性の高い舗装**: 複雑な排他ロジックを排除し、シンプルで予測可能な挙動にすることで、Laravelを知っていればメンテナンスできる舗装されたコードを維持しています。

#### 5. ビジネス貢献
この戦略により、**インフラコストを月額1,000円に抑えたまま、120万PVのトラフィックに耐えうる高速レスポンスを実現**。ビジネス要件とリソースを天秤にかけ、営業利益率を最大化させる判断を下しました。

長年手入力に依存していた旅費精算システムは従業員の負担が大きく、入力ミスも散見されていた。特に交通手段の発着駅と料金の計算は担当者の手入力に依存していた。

交通系ICカード内に記録されている **バイナリデータ（サイバネコード等）** から利用駅や料金を特定するロジックの実装を担当。

- **ユーザー負担を極限まで減らすGUI設計・実装:**
ジョルダンAPIにて取得した発着駅の一連の情報を旅費精算の起票画面に分かりやすく一覧表示。新幹線などICカード利用外の交通手段についても、ジョルダンAPIからの自動入力機能や、ICカード情報との組み合わせ選択が可能な直感的でミスのないGUIを設計・実装。従業員が迷わず、効率的に入力できる仕組みを提供した。

#### **【課題：専門部隊も予期せぬ「不定期なセッション切断」】**
部内ツールの商用SaaS展開に向け、専門のインフラ部隊と合意した「Cloud Run + Load Balancer」構成で構築。しかし、本番リリース3日前、特定の条件下で認証が解除される原因不明の事象が発生しました。

#### **【闘争のプロセス：事実に基づく科学的な切り分けと決断】**

社内インフラ部隊が「インフラ側に異常なし」とし、アプリ側のロジックを疑う声が強まる中、私は「事実（ファクト）のみを信じる」スタンスを貫き、以下のロジックで原因を特定しました。

1. **環境比較による原因の切り分け（変数の特定）**:
    * **ローカル環境（正常）**：アプリの認証ロジック、コードそのものに不備はないことを確認。
    * **GCE：常駐インスタンス（正常）**：GCP上のDB、ネットワークに問題がないことを確認。
    * **Cloud Run：サーバーレス（異常発生）**：異常が起きるのはCloud Run環境のみ。つまり、サーバーレス特有の **「インスタンスの揮発性（消滅と生成）」および「オートスケール」** に原因があると仮説を立てる。

2. **執念の「F5連打」による仮説検証**:
    この仮説をインフラ部隊に説明したところ、「最小インスタンス数=1」の設定変更で解決すると断言された。しかし、本質的な解決になっているかを確認するため、自らブラウザで **F5連打による負荷再現** を実施。
    * **検証結果**：オートスケールにより2台目が立ち上がった瞬間にログアウトが発生する事実をメトリクスから突き止める。根本原因は「最小インスタンス数」の設定といった小手先の小細工では解決不能であることを物理的に証明。

3. **戦略的トレードオフによる技術選定（Redisの採用）**:
    解決にあたっては、セッション情報を外部化する「完全ステートレス設計」への刷新を即断。極限のタイムリミットの中で以下の比較検討を行いました。
    * **検討した選択肢（Firestore）**：公式ドキュメント等では中小規模向けのセッションストアとして推奨されるが、本番リリース3日前という状況では未知のリスクを排除すべきと判断。
    * **採用した選択肢（Cloud Memorystore for Redis）**：Spring Sessionとの親和性が極めて高く、最も確実に、かつ最短で実装・検証を完遂できる手段として採用を決定。
    * **テックリードとしての判断**：Spring Bootのエコシステムをフル活用することで、**わずか数行の実装で堅牢な基盤への刷新を実現**。自身の技術的エゴではなく、プロジェクトの完遂とリリース後の保守性を最優先しました。

**結果：**
致命的な不具合を解消し、一度の遅延もなくSaaSサービスをリリース。現在に至るまで、セッション起因のトラブルはゼロを継続しています。

**【技術的意思決定の背景】**
私は「環境の違いで挙動が変わるのは、アプリケーション側の不備である」と考えています。本件においても、ローカル環境・GCE・Cloud Runの3環境を比較検証し、サーバーレス特有の変数を特定するに至りました。インフラ側のスペック調整ではなく、Redisを用いたステートレスなアーキテクチャへの刷新を選択することで、環境に依存しない堅牢なシステムへと昇華させました。

#### **【決断：リリース3日前のアーキテクチャ刷新と本番稼働後の戦略的撤退】**
もはや「設定の調整」では解決不能と判断。プロジェクトを救うため、以下の2つの苦渋かつ合理的な決断を下しました。

- **インフラの正攻法（舗装）：** 
  リリース3日前、Spring SessionとRedis（Memorystore）を導入し、セッション情報を外部化する「完全ステートレス設計」への変更を即断・即実装。専門部隊も見落としていた「サーバーレスの急所」を、土壇場で力技で舗装し直しました。
- **戦略的デスコープ（縮退）：** 
  目玉機能であった「AI自動文字起こし」のパフォーマンスが商用基準に達していないと判断。不安定な機能を出すことはプロダクトの信頼を毀損するため、**あえて目玉機能を削り、堅牢な認証と基本機能のみでリリースする**と上層部に提言。機能縮退を断行しました。

#### **【ビジネス貢献】**
- **予定通りの本番稼働:** 致命的な不具合を解消し、一度の遅延もなくSaaSサービスをリリース。
- **インフラ管理コストの抑制:** 構成が複雑なK8s等へ逃げることなく、Cloud Runの低コストな運用性を維持したまま安定稼働を実現。
- **現在進行形での稼働:** 「AI機能削除」という決断により、コアバリューである「議事録管理・タスク抽出」のUXを損なうことなく、現在もトラブルゼロで運用中。

#### **【全体最適への昇華】**  
「AI機能のリリース」という目先の成果を追う部分最適を捨て、アーキテクチャの刷新と戦略的デスコープの決断により **「プロダクトの予定通りの本番稼働」** という全体最適を完遂した事例です。

クラウド全盛の現代において、リソース不足を「札束（スペックアップ）」で解決することは容易です。しかし、私は安易なスペックアップの前に「アプリケーションレベルで解決できることはないか？」を徹底的に問うスタイルを貫いています。

- **新規案件（特命）:** Python (Slack API) など、開発スピードとライブラリの充実度を考慮し、要件に最適な技術をゼロベースで選定。
「使える言語」に縛られず、「解決すべき課題」に合わせて即座に手札を増やすスタイルです。

#### **チームにおける私の役割：「攻め」のための「守り」と「舗装」**
モダンな開発プロセス（スクラム、CI/CD、高速デプロイ）は大歓迎です。それらを推進する若手やミドル層の勢いを止めるのではなく、むしろ加速させることが私のミッションです。

- **主役を支える：** 1日10回デプロイする開発者が主役なら、私はその舞台（基盤）が壊れないよう補強し、デプロイの障壁となる技術的課題を先回りして解決します。
- **ボールを拾う：** 速度を優先するあまり取りこぼされた複雑なバグや、放置されたパフォーマンスの劣化、設計の歪みを、ソースコードベースで執念深くチューニングします。
- **秩序ある突破：** 停滞を打破するために「空気を読まない進言」もしますが、それは常に組織の秩序（上長やリーダーとの事前合意）を重んじた上での戦略的なアクションです。

「綺麗なコードを書くこと」以上に、「ビジネスが止まらず、チームが迷わず、価値を届け続けられる状態」を作ること。それが私のテックリードとしての矜持です。

#### **組織への向き合い方：「誠実な客観性」という武器**
私はプロジェクトに対して常に『誠実』でありたいと考えています。

支援として参画した際、現場のリーダーが疲弊し、危機感を感じつつも上層部へのエスカレーションができない状況を目の当たりにしました。私は **『支援者という客観的な立場』を戦略的に活用**し、現場の忖度や疲弊に飲み込まれることなく、あえて空気を読まない『幽霊担当者入りのWBS』を提示しました。
これは、単なる独断専行ではなく、疲弊した現場に代わって事実を突きつけることが、プロジェクトとメンバーを救う唯一の手段だと判断したからです。この『越境する力』こそが、私の考えるテックリードの責務です。

しかしながら、私の突破力は、周囲の協力があって初めて成立するものです。だからこそ、周囲への適切な相談と、透明性の高いエスカレーションを、技術スキルと同じくらい大切にしています。
最終的な意思決定に従う柔軟性と、目的達成のために粘り強く対話を続ける姿勢。
この両輪で、不確実性の高いプロジェクトを確実に完遂させます。

## 🛣️ 私が「舗装屋」を名乗る理由

これまで20年間、仕様なき解析、極限の最適化、炎上案件の救済など、数多くの「泥臭い現場」を独力で突破してきました。その時、常に感じていたのは、 **「今この場面で、私でない誰かが事前に道を整えてくれていたら、チームはどれほど速く前に進めただろうか」** という切実な思いでした。

本来、エンジニアリングは、もっと創造的で、ワクワクするものであるべきです。
私自身、10KBの死闘やキャッシュ・スタンピードとの戦いは、その瞬間は苦しくとも「どう制約を突破するか」を考えている時間は最高に楽しく、思い通りに動いた時の達成感は格別でした。

一方で、現実のプロジェクトには、誰もやりたがらない、しかし誰かがやらなければ進めない「泥」が至る所に転がっています。

私は、自分が培ってきた技術力と執念を、自分一人のためではなく **「次に続くエンジニアが、迷わず、全速力で駆け抜けるための道を作る」** ために使いたいと考えています。

## ⚙️「使われにいく」テックリードとして

私が目指すのは、高みから指示を出すリーダーではありません。現場でバリバリと新機能を実装するメンバーに寄り添い、**「今、何に困っているか」「どこを自動化すればもっと加速できるか」を徹底的にヒアリングし、自ら泥の中に手を突っ込んで道を舗装する役目**です。

*   **「デプロイ待ちの時間が集中力を削いでいるなら、数分で終わるようにパイプラインを組み直す」**
*   **「不具合調査に時間がかかっているなら、即座に原因を特定できる監視基盤を整える」**
*   **「誰も拾わないが、放置すると後で火を吹くボールを、ソースコードレベルで密かに処理しておく」**

どんな些細なストレスでも、現場の「負」を吸い上げ、技術でそれを「無かったこと」にする。
41歳のベテランとしてではなく、**チームが最高速を出すための「最も使い勝手の良い高機能なパーツ」** として、現場の不条理を削り、全員が全力で走れる環境を整える。

それこそが、私が20年のキャリアを経て辿り着いた、テックリードとしての矜持です。
